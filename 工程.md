# GuJumpgate iCloud API 工程手册

本文记录当前工程的正确使用方式、关键命令、外部兑换对接方式，以及这轮开发中已经踩过并处理过的问题。以后遇到类似日志，优先按本文排查。

## 1. 当前工程定位

这是一个 Chrome MV3 扩展，用侧边栏驱动 OpenAI / ChatGPT 注册、iCloud API 邮箱池轮询、ChatGPT AC 同步、AC 资格检查、外部 CDK 兑换队列监听。

核心入口：

- 扩展入口：`manifest.json`
- 后台主逻辑：`background.js`
- 自动执行控制：`background/auto-run-controller.js`
- 步骤 4 获取注册验证码：`background/steps/fetch-signup-code.js`
- 认证页内容脚本：`content/signup-page.js`
- 侧边栏：`sidepanel/sidepanel.html`、`sidepanel/sidepanel.js`
- 外部兑换本地代理：`tools/external-redeem-proxy.js`

## 2. 每次修改代码后的固定动作

修改扩展代码后必须重新加载扩展：

1. 打开 `chrome://extensions`
2. 找到 `GuJumpgate iCloud API`
3. 点击重新加载
4. 重新打开目标 Chrome 窗口侧边栏

原因：Chrome 扩展的 service worker、content scripts、DNR 规则不会自动使用磁盘上的新代码。

语法检查命令：

```bash
cd /Users/lyr/Desktop/GuJumpgateIcloudApi
node --check background.js
node --check background/steps/fetch-signup-code.js
node --check background/message-router.js
node --check sidepanel/sidepanel.js
node --check tools/external-redeem-proxy.js
```

## 4. 外部兑换当前设计

### 4.1 为什么改成本地 Node 代理

之前扩展直接请求远端接口时出现过：

```text
HTTP 403
body=Invalid CORS request
```

随后又出现：

```text
HTTP 403
code=10003
message=CSRF verification failed
```

Postman 同样 body 和 API Key 可以成功，说明问题不是 CDK、AC 或 API Key，而是浏览器请求环境带来的 `Origin` / CORS / CSRF 差异。

现在的请求链路是：

```text
Chrome 扩展
  -> http://127.0.0.1:18789/external-redeem
  -> Node fetch
  -> https://chong.nerver.cc/api/external/...
```

Node 请求不会携带 `chrome-extension://...` 的浏览器 Origin，因此不会触发浏览器侧 CORS/CSRF 问题。

### 4.2 本地代理限制

本地代理不是通用 HTTP 代理，只允许两个远端路径：

```text
/api/external/cdkey-redeems
/api/external/cdkey-redeems/status
```

这样避免本地服务被误用成任意转发器。

### 4.3 提交兑换 curl

手动验证提交：

```bash
curl -i -X POST 'https://chong.nerver.cc/api/external/cdkey-redeems' \
  -H 'accept: application/json' \
  -H 'content-type: application/json' \
  -H 'X-External-Api-Key: <你的API_KEY>' \
  --data-raw '{"items":[{"cdkey":"PYAW-NTSL-WX4E-QDB5","access_token":"<AC_TOKEN>"}]}'
```

不要加这些浏览器头：

```text
Origin
Referer
sec-fetch-*
cookie
X-CSRF-*
```

### 4.4 查询当前进度 curl

状态查询按 `cdkeys` 查，不按 `task_id` 查：

```bash
curl -i -X POST 'https://chong.nerver.cc/api/external/cdkey-redeems/status' \
  -H 'accept: application/json' \
  -H 'content-type: application/json' \
  -H 'X-External-Api-Key: <你的API_KEY>' \
  --data-raw '{"cdkeys":["PYAW-NTSL-WX4E-QDB5"]}'
```

成功提交但还没兑换完成时，常见状态：

```json
{
  "status": "pending_dispatch",
  "display_status": "等待充值",
  "accepted": true
}
```

### 4.5 扩展里的自动兑换流程

每轮注册成功后：

```text
注册成功
  -> 自动读取 https://chatgpt.com/api/auth/session
  -> 提取 accessToken / 邮箱 / 账号信息
  -> 调用资格检查接口
  -> eligible === true 且 coupon_state === "eligible"
  -> 取 CDK 池下一个未用 CDK
  -> 通过本地 Node 代理提交兑换
  -> 写入外部兑换队列
  -> alarm 定期查询状态
```

资格判断使用严格条件：

```js
token_ok === true
status === 200
eligible === true
coupon_state === "eligible"
```

## 5. 飞书多维表格同步

### 5.1 不使用浏览器 Cookie

飞书同步不要使用浏览器 Cookie。Cookie 会过期、会泄露登录态，也不适合放进扩展配置。

当前实现只使用飞书开放平台应用凭证：

```text
App ID
App Secret
多维表格 App Token
Table ID
```

本工程当前已经内置默认飞书配置，侧边栏会自动带出：

```text
App ID: cli_aaa96e6d0c385be9
多维表格 Wiki Token: TElOwYihkia8ESkui8Gc1yXTn5c
Table ID: tblFw7TwPiv17eRv
```

当前多维表格链接是 wiki 链接：

```text
https://qcn5tem5vf41.feishu.cn/wiki/TElOwYihkia8ESkui8Gc1yXTn5c?table=tblFw7TwPiv17eRv&view=vewsvkxUmt
```

后台会先通过飞书 Wiki 节点接口把 `TElOwYihkia8ESkui8Gc1yXTn5c` 解析成真实多维表格 `obj_token`，再调用多维表格 records API。

后台会通过：

```text
POST https://open.feishu.cn/open-apis/auth/v3/tenant_access_token/internal
```

换取 `tenant_access_token`，再调用多维表格记录接口。

### 5.2 表格字段

先在飞书多维表格里手动创建这些字段，字段名必须一致：

```text
邮箱
执行状态
失败原因
AC资格
AC检测原因
CDK
兑换状态
更新时间
```

推荐字段类型：

```text
邮箱 / 执行状态 / 失败原因 / AC资格 / AC检测原因 / CDK / 兑换状态：文本
更新时间：日期
```

### 5.3 同步时机

侧边栏开启 `每轮结束后同步结果` 后，以下节点会自动同步：

```text
账号轮次结束
AC 读取和资格检查完成
外部兑换提交成功或失败
外部兑换队列状态刷新
```

同步逻辑按 `邮箱` upsert：

```text
飞书表中已有该邮箱 -> 更新这一行
飞书表中没有该邮箱 -> 新增一行
```

飞书同步失败不会中断注册、AC 检测、兑换队列，只会写入日志和侧边栏 `飞书多维表格同步` 状态。

### 5.4 权限要求

飞书应用需要能访问目标多维表格，并具备读取和写入记录权限。配置完成后先点侧边栏 `测试连接`，测试只验证 token 和表格读取权限，不会写入测试行。

### 4.6 CDK 池和失败记录规则

侧边栏 `外部兑换队列` 区域新增两个 CDK 操作：

- `上传 CDK`：选择 `.txt` / `.csv` 文件，将文件里的 CDK 合并到当前 CDK 池并自动去重。换行、逗号、分号、Tab、空格都可以作为分隔符。
- `清理 CDK`：清空 CDK 池，并删除本地 `submit_failed` / `rejected` / `not_found` 这类没有进入外部队列的失败记录。

注意：`HTTP 403`、`CSRF verification failed`、`Invalid CORS request` 这类提交失败，通常代表浏览器请求被拦截，CDK 不一定已经被消耗。现在代码只会把外部接口已接受的记录算作 CDK 占用：

```text
accepted=true
或存在 task_id
```

本地提交失败记录不会再占用 CDK。已经进入外部队列的记录会保留，继续由 `刷新队列` 或定时 alarm 查询状态。

### 4.7 充值失败重试

外部兑换队列里 `提交失败` / `充值失败` / `兑换超时` 等终态失败记录会显示 `重试` 按钮。

当前重试机制：

```text
点击重试
  -> 按队列项邮箱查找本地已同步的完整 AC
  -> 使用原 CDK + 本地 AC 重新提交外部兑换
  -> 成功后原队列项更新为新的 task / 等待充值状态
  -> 进入定时查询
```

如果本地没有该邮箱的完整 AC，会提示先登录该账号并同步 AC。后续如需全自动“重新登录账号 -> 获取 AC -> 再提交兑换”，需要接入 OAuth 登录链路。

## 5. ChatGPT AC 使用方式

侧边栏 `ChatGPT AC` 区域：

- `同步 AC`：手动读取当前 ChatGPT session 并检查资格
- `查看`：展开所有已同步 AC 记录，按当前账号优先、其它账号按更新时间倒序展示
- `导出 Excel`：导出邮箱、AC、是否有资格、CDK 四列；CDK 按邮箱匹配外部兑换队列里最新一条记录

自动执行时，不需要手动点 `同步 AC`。每轮注册成功后会自动同步 AC、检查资格、进入外部兑换逻辑。

注意：

- AC 是敏感信息，不要写进日志和文档。
- 前端默认只展示预览，完整 token 仅用于提交兑换。
- 如果要生成手动 curl，在 DevTools 里从扩展 storage 临时读取即可，不要落盘。

## 6. iCloud API 邮箱池规则

邮箱池按顺序轮询。每个邮箱成功使用后会自动标记 `已用`。

按钮含义：

- `保存配置`：保存当前文本框邮箱池
- `清空已用`：把已用状态清掉，让现有邮箱重新变成未用
- `清空全部`：清空整个 iCloud API 邮箱池
- `恢复默认邮箱`：恢复默认配置

历史问题：

```text
自定义邮箱池为空，请先至少填写 1 个邮箱。
```

原因：邮箱池文本框为空，或者保存后状态里没有可用邮箱。

```text
自定义邮箱池第 2 个邮箱不存在，请检查邮箱池数量是否与自动轮数一致。
```

原因：自动轮数大于可用邮箱数量。例如跑 2 轮，但只有 1 个未用邮箱。

处理：

1. 补够邮箱
2. 点击 `保存配置`
3. 必要时点 `清空已用`
4. 再跑自动执行

## 7. 步骤 4 验证码相关坑

### 7.1 等待注册验证码页面就绪超时

典型日志：

```text
步骤 4：等待注册验证码页面就绪超时，90 秒内未拿到内容脚本 PREPARE_SIGNUP_VERIFICATION 回复。
```

这个错误不等于“邮箱接口没验证码”。它通常表示背景页和认证页 content script 没通信成功。

常见原因：

- 认证页正在跳转
- 页面进入 Chrome 错误页
- 页面被 Cloudflare / OpenAI retry 页拦住
- content script 未注入或扩展没重新加载
- 标签页被刷新到非 OpenAI auth 域名

现在日志会带 tab 快照：

```text
tab=<id> status=<status> url=<url>
```

优先看 URL 是否还是：

```text
auth.openai.com
auth0.openai.com
accounts.openai.com
```

### 7.2 验证码暂未获取到时不回到 open-chatgpt 重开

之前的问题是步骤 4 没等到验证码后会回到 `open-chatgpt` 重开，导致验证码页被刷新，刚发出的验证码衔接不上。

当前处理：

```text
验证码暂未获取到 -> 保持当前验证码页 -> 不回到 open-chatgpt 重开
```

这样便于人工刷新认证页或继续等待邮箱接口。

### 7.3 需要更多定位日志的位置

已经加过或保留的关键日志点：

- 步骤 4 准备验证码页
- 内容脚本通信异常
- 认证 retry 页恢复
- iCloud API 邮箱轮询
- AC 同步
- 外部兑换提交
- 外部兑换状态查询
- 填写姓名 / 年龄后的页面稳定等待

### 7.4 步骤 2 内容脚本超时误报

曾出现日志：

```text
失败：认证页 内容脚本 20 秒内未响应，请刷新页面后重试。
```

但页面实际上已经进入 `检查你的收件箱` / 验证码页。这说明邮箱填写和点击动作已经完成，只是后台没有及时收到内容脚本返回。

当前处理：

- `submit-signup-email` 的后台响应等待提高到 60 秒
- 内容脚本不再用延迟 `setTimeout` 异步点击，而是点击完成后再返回后台
- 如果后台仍收到通信超时，会检查当前 tab 是否已进入密码页 / 邮箱验证码页 / 手机验证码页 / 资料页；若已进入后续页面，则按步骤 2 成功继续

## 8. auth.openai.com HTTP ERROR 500 自动刷新

遇到 Chrome 错误页：

```html
auth.openai.com 目前无法处理此请求。
HTTP ERROR 500
```

现在步骤 4 会自动识别并刷新当前认证 tab。

识别条件包括：

- Chrome neterror 页面
- `main-frame-error`
- `HTTP ERROR 500`
- 文案包含 `auth.openai.com`

处理逻辑：

```text
检测到 auth.openai.com HTTP 500 网络错误页
  -> 不调用 chrome.tabs.reload，避免 Chrome 弹出“确认重新提交表单”
  -> chrome.tabs.update(tabId, { url: 当前 auth.openai.com 地址 })
  -> 这相当于重新发起一次 GET 导航，接近后端 POST-Redirect-GET 的效果
  -> 等待页面稳定后再次检查
  -> 若页面已恢复，继续验证码页准备/提交结果确认
  -> 若恢复后内容脚本仍无响应，切回 https://chatgpt.com/auth/login
  -> 若 GET 导航后仍是 HTTP 500，则切回 https://chatgpt.com/auth/login
  -> 抛出 AUTH_HTTP_500_RELOGIN_CURRENT_ACCOUNT，由自动执行沿用当前邮箱回到 open-chatgpt 重新登录
```

验证码提交后的额外恢复逻辑：

```text
提交验证码后认证页通信断开
  -> 先读取一次 tab / DOM 快照
  -> 未命中则等待约 800ms 再读取一次
  -> 若识别到 auth.openai.com HTTP 500，用 GET 导航重新打开当前认证地址
  -> 若当前仍停留在 auth.openai.com/email-verification，也用 GET 导航当前地址，避免 POST 表单重复提交弹窗
  -> GET 导航恢复后只重新提交同一个验证码一次
  -> GET 导航后仍不可恢复，或恢复后内容脚本仍无响应时，切回登录入口并沿用当前账号重登
```

日志：

```text
步骤 4：检测到 auth.openai.com HTTP 500 网络错误页，不刷新 POST 表单结果页，正在用 GET 导航重新打开认证地址：...
步骤 4：GET 导航后仍是 auth.openai.com HTTP 500，正在切回登录入口以重新登录当前账号。
```

## 9. Cloudflare 风控处理

工程里已有 Cloudflare 风控检测，不是绕过。

典型触发：

```text
max_check_attempts
CF_SECURITY_BLOCKED
```

当前策略：

- 一旦触发 Cloudflare 安全防护，停止流程
- 不再连续刷新
- 不再反复点重试
- 提示等待 15-30 分钟或更换浏览器

不要在这种状态下频繁点重新发送验证码、刷新认证页、反复重试；这会加重风控。

### 10.2 本地代理不可用

外部兑换本地代理不可用，请先启动 Node 代理

cd /Users/lyr/Desktop/GuJumpgateIcloudApi

screen -dmS gujumpgate-external-redeem-proxy npm run external-redeem-proxy

## 11. 自动执行多轮规则

多轮自动执行时，每一轮都应该重新从流程起点开始，并使用下一个邮箱。

当前关注点：

- 轮数要小于等于可用邮箱数量
- 每轮成功后自动同步 AC
- 每轮成功后自动检查资格
- 资格通过后提交外部兑换
- 外部兑换进入队列后由 alarm 监听，不阻塞后续状态查询

失败轮次日志会记录：

```text
失败轮次：第 N 轮（重试 X 次，最终原因：...）
```

遇到失败先看最后一个失败节点，不要只看汇总。

## 12. 第 2 步内容脚本超时与已登录误判

这次遇到的典型日志：

```text
失败：认证页 内容脚本 20 秒内未响应，请刷新页面后重试。
步骤 2：检测到当前停留在已登录 ChatGPT 首页，已阻止自动跳过步骤 3/4/5。
```

但页面实际是 `chatgpt.com/` 登录/注册入口时，不应该被当作已登录首页。已修正：

- `chatgpt.com/` 根页不再单凭 URL 判定为已登录首页
- 第 2 步提交邮箱/手机号的后台等待统一拉长
- 入口页、密码页、内容脚本内部等待不再使用 20 秒短窗口
- 如果提交后页面已经进入验证码页，会按步骤 2 成功继续，不再回头重开

修改扩展后必须重新加载插件，并刷新/重开当前认证页，否则旧 content script 仍可能报旧的 20 秒错误。

## 13. 自动任务窗口不可用

典型错误：

```text
自动任务窗口已不可用，请在目标 Chrome 窗口重新打开侧边栏并启动任务。原因：No window with id: 0.
```

原因：

- 启动自动任务的 Chrome 窗口关闭了
- Chrome windowId 过期
- service worker 里保留了旧窗口上下文

处理：

1. 在目标 Chrome 窗口重新打开侧边栏
2. 重新点击 `自动执行`
3. 必要时重新加载扩展

代码里已经对部分窗口失效情况做了 fallback 和清理，但目标窗口关闭后仍应重新从目标窗口启动。
